## 論点1：HEIC変換

〔論点〕
iPhoneのカメラ写真はHEIC形式（Apple独自の画像のいれもの）で入ってくることがある。HEICはJPG/PNG前提の場所では開けない・表示できないため、本番でこの写真をどこで・いつJPEGへ変換するか、そして変換に失敗した場合どうするかを決める必要がある。

モックの実装事実：写真取り込み時に `createImageBitmap` で画像を読み、canvasへ描き直し、`toDataURL("image/jpeg", 0.72)` でJPEGへ焼き直している（長辺1280pxへ縮小）。形式が何であれ取り込んだ瞬間にブラウザ内でJPEG化される作り。失敗時は `FileReader.readAsDataURL` で元ファイルをそのまま保存するフォールバックが入っている。

〔方式A：モック完全踏襲（canvas変換 → 失敗時は元ファイルをそのまま保存）〕
モックのコードをそのまま本番へ移す。canvas変換に失敗したらHEICのまま保存する。
- 利点：モックと1行も違わない。実装が最も速い。
- 欠点：変換失敗時、HEICのまま保存された写真はビューアで表示できず「開けない写真」がストーリーに残る。変換したフリをして壊れた写真を保存する挙動になり、ユーザーには原因が分からない。

〔方式B：canvas変換 → 失敗時はその写真を保存せず、失敗を知らせてスキップ（推奨）〕
変換のしくみはモックと同じ（取り込み時にcanvasでJPEG化）。ただし変換に失敗した写真は保存せず、失敗したことをユーザーに知らせてスキップする（通知の文言は全体仕様書§17で規定する。本書は文言を持たない）。
- 利点：ストーリーに「開けない写真」が絶対に残らない。ユーザーに失敗が伝わる。変換のしくみ自体はモックと同一なので挙動の見た目は変わらない。
- 欠点：モックのフォールバック1か所だけ挙動が変わる（ただしこれは"壊れた保存"を防ぐ修正であり、モックの正しい意図に沿う）。

〔決定〕方式B（canvas変換 → 失敗時は保存せず、失敗を知らせてスキップ。通知の文言は全体仕様書§17が持つ）

〔理由〕
モックのフォールバック（元ファイルをそのまま保存）は、変換できなかったHEICを生のまま保存するため、実質的に保険として機能せず「表示できない写真がたまる」最悪の結果を招く。canvas変換のしくみ自体はモックと同一に保つため、ユーザーから見た取り込み挙動は変わらない。変わるのは"失敗したときに壊れた写真を残さない"という一点のみで、これはモックの設計意図（取り込み時に必ず表示可能な形へ変換する）を正しく満たす方向の修正。この変換の最終確認は、iPhone実機を持つ片倉さんの実機検証で取る（前提資料の役割分担どおり）。

## 論点2：写真実データの保存方式

〔論点〕
編集・登録した写真の画像データそのものを、本番でどこに保存するか。ログイン不要・外部サーバー送信なしという企画前提（全体仕様書§3）を保ったまま、写真をためていける保存先を選ぶ。

モックの実装事実：`localStorage.setItem("tearose-data-v1", JSON.stringify(data))` で、画像をDataURL文字列として `photo.image` に入れ、photos/albums丸ごとをlocalStorageへ保存している。全体仕様書§20は「本番保存先はIndexedDBを想定」「保存件数に固定上限を設けない」と明記。§3は保存失敗時に容量超過トーストを出すと規定。

〔方式A：モックのままlocalStorageに保存〕
画像DataURLを含む全データをlocalStorageに持つ。
- 利点：モックと同一。
- 欠点：localStorageは数MB程度が上限で、DataURL化した写真は容量を食う。写真を数枚〜十数枚ためた時点で容量オーバーになり「保存容量がいっぱいです」が頻発する。「写真を日付ごとにためていく」という企画の中心機能が成り立たない。

〔方式B：IndexedDBに画像を保存（推奨）〕
画像実データはIndexedDB（同じ端末内の大容量ストレージ）に保存する。photos/albumsのメタ情報の保存先は実装時に確定（IndexedDB内に統一するのが素直）。いずれも端末内のみで、外部サーバーには一切送らない。
- 利点：写真を多数ためても容量オーバーになりにくい。「ためていく」企画に耐える。全体仕様書§20の本番想定に一致。
- 欠点：モックのlocalStorage実装からストレージ層を書き換える必要がある（ただし画面・データモデルは不変）。

〔決定〕方式B（IndexedDBに画像を保存）

〔理由〕
localStorageもIndexedDBも「端末内のみ・外部に送らない」点は同じで、企画の"公開しない"前提はどちらでも守られる。違いは容量のみ。本アプリは写真を日付ごとに蓄積し続けるのが中心機能であり、数MB上限のlocalStorageでは早期に容量オーバーで機能不全になる。全体仕様書§20が本番IndexedDBを想定し「保存件数に固定上限を設けない」と明記していることとも一致する。容量超過トースト（§3）は、IndexedDBでも保存失敗時のセーフティとして維持する。データモデル（§16 Photo/Album/Overlay型）は変更しない。

## 論点3：状態管理（ためる↔見せるの切替と永続化範囲）

〔論点〕
「写真をためる（ストーリー）」と「見せたい写真を並べる（アルバム）」の切替、および編集中の一時状態を、本番でどう保持するか。また、その状態のうち何を保存（永続化）し、何を画面を閉じたら消える一時状態とするか。

モックの実装事実：画面の切替は単一のReact state（`n`：story/albums/settings/picker/editor/viewer 等）で管理。アルバム編集内のタブ切替（再生リスト↔ストーリーからえらぶ）は別state（`tab`：playlist/choose）。まとめて編集モードや選択中の写真ID群も画面ローカルのstateで保持。永続化（保存）されるのは `data`（photos + albums）のみで、画面位置・選択状態・編集途中の一時状態は保存されない。バッチ編集中に端末「戻る」を押すとセッションごとリセットされる（全体仕様書§8.3）。

〔方式A：モックの状態管理を踏襲（画面・選択はメモリ上のstate、永続化はphotos/albumsのみ）〕
モックと同じく、画面遷移・タブ・選択状態はReactのstateで持ち、保存対象はphotos/albumsに限る。
- 利点：モックの挙動（戻るでリセット、再読込でストーリー/アルバムの中身は残るが画面位置は初期化）を完全に再現する。全体仕様書§8.3の「戻るで全リセット」もそのまま満たす。
- 欠点：特になし（モックの挙動が正本であるため）。

〔方式B：画面位置や選択状態も永続化する〕
どの画面を開いていたか、選択中の写真は何かまでを保存し、再読込後に復元する。
- 利点：再読込しても直前の画面に戻れる。
- 欠点：モックにない挙動を創作することになる（全体仕様書§0.1 補完禁止に抵触）。§8.3の「戻るで全リセット」と矛盾するリスク。モックが正本である以上、挙動を勝手に足せない。

〔決定〕方式A（モックの状態管理を踏襲。永続化はphotos/albumsのみ）

〔理由〕
モックは凍結済み構造の正本であり、状態管理の挙動（画面・選択はメモリ上、保存はphotos/albumsのみ、戻るでバッチ編集リセット）はモックの実装と全体仕様書§8.3に明記された確定挙動。方式Bは再読込後の復元というモックに無い挙動を足すことになり、補完禁止（§0.1）と§8.3に反する。したがって状態管理はモックをそのまま再現し、新たな判断を加えない。永続化する状態はphotos/albums（＝§16のデータモデル）に限定する。

## 論点4：ビルド出力先 / Pagesのpath / Actions定義

〔論点〕
本番はReact＋Vite構成をビルドし、GitHub Pagesへデプロイする（前提資料でActions自動デプロイが決定済み）。ビルドの出力先、Pagesで公開するpath、Actionsの起動条件という「デプロイの骨格」を決める。モックは単一HTMLのためこの情報を持たず、本番で新たに確定する必要がある。

前提資料の既定事項：デプロイはActionsで自動。経路は「git push → Actions → Pages」の一本のみ（別ホスティング・手動デプロイは禁止＝指示書§2デプロイ障害時ルール）。

〔方式A：プロジェクトサイト構成（リポジトリ名付きサブパスで公開）〕
`https://<ユーザー名>.github.io/<リポジトリ名>/` で公開する。Viteの `base` にリポジトリ名を設定。Actionsは main ブランチへのpushで起動し、ビルド成果物（`dist/`）をPagesへ配置。
- 利点：GitHub Pagesの最も標準的な構成。専用ドメイン不要。前提資料の「push → Actions → Pages」一本経路にそのまま乗る。
- 注意：Viteの `base` 設定を誤ると、公開後に画像やJSが読めない（パスずれ）。この一点だけ実装時に検証項目として明記が必要。

〔方式B：ユーザー/組織サイト構成（ルートで公開）〕
`https://<ユーザー名>.github.io/` のルートで公開。専用のリポジトリ名（`<ユーザー名>.github.io`）が要る。
- 利点：`base` 設定がシンプル（ルート `/`）。
- 欠点：1アカウントにつき1つしか作れず、他プロジェクトとリポジトリ名が競合する。このアプリ専用にアカウント直下を占有する形になり、汎用性が低い。

〔決定〕方式A（プロジェクトサイト構成。Viteの `base` にリポジトリ名を設定、main push でActions起動、`dist/` をPagesへ）

〔理由〕
前提資料でActions自動デプロイ・単一経路が確定しており、方式Aが「git push → Actions → Pages」に素直に乗る標準構成。方式Bはアカウント直下を1プロジェクトで占有するため将来の別プロジェクトと競合し、このアプリ専用に縛る合理性がない。方式A採用時の唯一の落とし穴はViteの `base` パス設定ミスによる公開後のパスずれであり、これは実機で「公開URLを開いて画像・操作が正常表示されるか」を確認する検証項目として、開発フェーズ序盤に組み込む。デプロイ障害時はevidence-first-ops（証拠→原因→合意→コマンド）に従い、別経路・別ホスティングは提案しない。

## 論点5：マイルストーンの区切り（実装順）

〔論点〕
本番実装（Codexが行う）を、モックのどの機能から順に組むか。1回で全部作るのではなく、途中で実機確認できる区切りに分けることで、「押す・見る」で判定しながら進められる。区切りの決め方は、モックの機能どうしの依存関係（Aが無いとBが試せない、という順序）から逆算する。あやめは各区切りの完了を実機で確認して次へ進む。

モックの実装事実（依存関係）：
- 写真は「取り込み → 変換 → 保存 → 一覧表示」で初めて存在する。ここが土台で、これが無いと編集もアルバムも試せない。
- 編集（回転・テキスト・キャプション）は、写真が保存・表示できて初めて意味を持つ。
- アルバム（再生リスト・表紙・並び替え）は、ストーリー写真が存在して初めて作れる（アルバムはストーリー写真のIDを参照するだけ＝全体仕様書§2.2）。
- ビューア（日別再生・アルバム再生）は、写真とアルバムが揃って初めて再生対象がある。
- 設定（初期化・引き継ぎモック等）は他機能に依存しない独立機能。

〔方式A：機能の依存順に4区切りで刻む（推奨）〕
モックの依存関係に沿って、下流が上流に依存する順で刻む。各区切りの末尾に実機確認を置く。

```
M1 土台：取り込み→HEIC変換→IndexedDB保存→ストーリー日付別一覧
        ★ここでHEIC変換の実機検証（片倉さん）を必ず通す
   ↓（写真が存在しないと以降を試せない）
M2 編集：写真回転・テキスト（移動/拡大/回転/色）・キャプション・上書き保存・削除
   ↓（見せる写真が仕上がって初めてアルバムが要る）
M3 アルバム：新規作成・再生リスト編集・ストーリーからえらぶ・表紙・並び替え
   ↓（写真とアルバムが揃って初めて再生対象がある）
M4 ビューア＋設定：日別/アルバム再生・自動再生・スワイプ / 設定の初期化・引き継ぎモック
        ★最後に公開URLでの表示（Vite baseパスずれ）を実機確認
```

- 利点：各区切りが「前の区切りが動かないと試せない」自然な順で、実機確認が空振りしない。HEIC変換の実機検証を最初のM1に置けるため、最大リスクを序盤で潰せる（指示書P5「HEIC変換の実機検証を序盤に必ず組み込む」に合致）。
- 欠点：特になし（モックの依存構造そのまま）。

〔方式B：画面単位で刻む（画面ごとに1区切り）〕
ストーリー画面・写真選択・写真編集・アルバム一覧…と画面ごとに区切る。
- 利点：区切りが画面と1対1で分かりやすい。
- 欠点：画面単位だと依存関係を跨ぐ。例：写真編集画面だけ先に作っても、保存先（IndexedDB）や取り込みが無ければ実機で試せず、実機確認が空振りする。HEIC変換の検証タイミングも後ろにずれ、最大リスクの発見が遅れる。

〔決定〕方式A（機能の依存順に M1土台 → M2編集 → M3アルバム → M4ビューア＋設定 の4区切り。M1でHEIC変換の実機検証、M4末尾で公開URL表示を実機確認）

〔理由〕
アルバムはストーリー写真のIDを参照する構造（§2.2）であり、写真の取り込み・保存・表示が無ければ編集もアルバムも実機で試せない。したがって依存の上流（土台）から順に刻むのが、各区切りの実機確認を空振りさせない唯一妥当な順序。画面単位（方式B）は依存を跨ぐため実機確認が成立しない区切りが生じる。特にHEIC変換は本アプリ最大の技術リスクで、指示書P5が「実機検証を序盤に必ず組み込む」と定めているため、M1（最初の区切り）で片倉さんの実機検証を通す配置とする。この実装順に沿ってCodexへ発注し、各Mの完了はあやめが実機で「押す・見る」で確認して次Mへ進む。
