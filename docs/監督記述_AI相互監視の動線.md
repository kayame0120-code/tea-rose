# 監督記述 — AI相互監視の動線

- 版：v0.1（P4準備）／作成：2026-07-21
- 目的：実装AI（Codex）の成果を、監督AI（Claude）が正本に照らして検品する動線を定める。
- 原則：**実装したAIと検品するAIを分ける。** 実装者の「できました」を完了の証拠にしない。

---

## 登場する2つの役割

| 役割 | 担当 | やること |
|---|---|---|
| 実装AI | Codex | 発注書を読み `src/` を実装。受け入れ条件YES/NO表を自己記入して報告。 |
| 監督AI | Claude | Codexの報告を**正本と照合**して検品。合否表を出す。合格判定はあやめの承認まで確定しない。 |

人間（あやめ）は、監督AIの合否表を見て、実機で「押す・見る」で最終確認し、次のマイルストーンへ進む宣言をする。

---

## 動線（1マイルストーンごとに1周する）

```
① あやめ：発注書 Mx を Codex に渡す（＝Codex投入。ゲート解禁後のみ）
        ↓
② Codex：src/ を実装 → git push（Codexがコマンドを出す）
        ↓
③ Codex：受け入れ条件YES/NO表を自己記入 → 各項目に根拠（ファイル名・行）を添えて報告
        ↓
④ Claude：報告を鵜呑みにせず、正本と照合して検品（下記「検品の3点」）
        ↓
⑤ Claude：合否表を出す（合格 / 修正 / 却下）。NG項目は「どの正本のどこに反する」を明記
        ↓
⑥ あやめ：実機で押す・見るで確認 → 合格なら次マイルストーンへ進む宣言
```

- ④で不合格が出たら②へ戻す（Codexが修正 → 再push → 再報告）。
- **Claudeは合否を出すが、確定はしない。** 進行の宣言はあやめのタップのみ。

---

## 検品の3点（Claudeが④で必ず見る）

### 点1：受け入れ条件YES/NO表の照合
- Codexが「YES」と書いた各項目を、Claudeが正本（モック・全体仕様書）と突き合わせて**独立に判定し直す**。
- Codexの根拠（ファイル名・行）を実際に見る。根拠が示されていない項目は自動的に「未確認」扱い。

### 点2：デザイントークン照合
- 色・余白・角丸・フォントが、モックの `:root` トークンと既存CSSクラスに**一致しているか**を確認する。
- Codexが独自の色値・余白を入れていたら不合格（AGENTS.md §2「UIを新規デザインしない」違反）。
- 照合対象トークン（モック `:root`）：`--cream #FAF6F2` / `--powder #F4E7E4` / `--rose #D8A5A2` / `--beige #EBE1D5` / `--tea #4A3A32` / `--sage #A4B6A0` / `--night #2A211E`。
- 文字色パレット14色（全体仕様書§8.6）、主要ボタン地色（§25.4）も照合対象。

### 点3：文言の一字一句照合
- ボタン名・見出し・トースト文・説明文が、全体仕様書に書かれた文字列と**完全一致**するか。
- 言い換え・意訳・句読点の違いも不合格。特に `写真をとって あげる`（スペース含む・変更禁止）等。

---

## Claudeが証拠なしで合格を出さないための自己ルール

- Codexの報告文だけで合格にしない。**実際のコード／正本を見た項目だけ**を判定する。
- 見ていない項目は「未確認」と明記する。「たぶん合っている」で埋めない。
- 正本どうしが矛盾していたらモックを正とし、矛盾自体をあやめに報告する（自己解決して混ぜない）。

---

## この監督記述が守っていること（根拠）

- 指示書§1-6：Codexの完了報告を完了と認めない。YES/NO表＋トークン照合で検品し、証拠のない項目は「未確認」と明記。
- 指示書§4（P4）：AI同士で監視する動線を組む。検品までP4で行う。
- 前提資料：実装（src/）はCodex、コード検品はClaude、進行宣言はあやめ。
