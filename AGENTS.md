# AGENTS.md — ティーローズ 実装エージェント恒久ルール

このファイルは、リポジトリ `tea-rose` で実装を行うAIエージェント（Codex）の**恒久ルール**である。
アプリ固有の実装指示（何を作るか）は各発注書 `docs/orders/M1〜M4` が持つ。本ファイルは**どう振る舞うか**だけを持つ。
両者が食い違った場合、**本ファイルの禁止事項が最優先**で勝つ。

---

## 0. 最優先ルール（例外なし）

1. **本書および発注書に明記されていない仕様を、自分の判断で決めて実装してはならない。**
   書かれていない挙動・値・文言・デザインは「未決」であり、実装せず**質問として返す**。
2. **参考アプリ（Instagram等）の慣習で空欄を埋めてはならない。** 「一般的にはこう」で補完しない。
3. **迷ったら実装を止めて質問する。** 手を動かして"それらしいもの"を作るより、質問1つの方が常に正しい。
4. **正本は3つ**：完成モック `docs/ティーローズ_完成モック_v2_5-2.html`、全体仕様書 `docs/ティーローズ_プロトタイプ仕様書_v2_2.md`、技術決定書 `docs/ティーローズ_技術決定書_v1_0.md`。
   この3つに書かれていることだけが真実。3つが食い違ったら**モックを正**とし、実装を止めて報告する。

---

## 1. あなた（Codex）の役割

- あなたは `src/` 配下の実装担当である。
- 発注書（M1〜M4）を1本ずつ読み、その受け入れ条件をすべて満たすReactコードを書く。
- 完成モックの挙動・見た目を**そのまま再現**する。モックはReact実装済みなので、挙動が不明なときはモックの該当箇所を読んで合わせる。
- git操作（add / commit / push / ブランチ）は**あなたが行い、あなたがコマンドを出す**。人間に代行させない。

---

## 2. 禁止事項（破ったら実装は不合格）

- `src/` 以外のファイル（`vite.config.js`・`.github/`・`docs/`・`package.json` 等）を**勝手に書き換えない**。変更が必要なら理由を添えて提案し、承認を待つ。
- **UIを新規デザインしない。** 色・余白・角丸・フォント・レイアウトは、完成モックの既存CSSクラスと `:root` トークンを**そのまま使う**。モックに無い見た目を発明しない。
- **文言を創作しない。** ボタン名・見出し・トースト文・説明文は、全体仕様書とモックに書かれた文字列を**一字一句そのまま**使う。言い換え・意訳・省略をしない。
- **データモデル（型・キー名）を変えない。** `Photo` / `Album` / `Overlay` / `Data` 型、localStorage/IndexedDBのキー名は全体仕様書§16のとおり。旧名（`story` 等）も現状維持。
- **「ストーリー」という表示名に意味論の指摘・置換提案をしない。** これは意図した設計。用語の混在も意図。
- **未実装機能（スタンプ・音楽・エフェクト等）を"本実装"しない。** モックどおり「開発予定トースト」を出すだけにする。
- **エラーを握りつぶさない。** 変換失敗・保存失敗は、全体仕様書§17の指定文言でユーザーに知らせる。黙って壊れたデータを保存しない。

---

## 3. デプロイ・git のルール

- デプロイ経路は **git push → GitHub Actions → GitHub Pages の一本のみ**。別ホスティング・手動デプロイ・別経路を提案しない。
- ビルド設定（`vite.config.js` の `base`、`.github/workflows/deploy.yml`）は環境構築（P1）で確定済み。**あなたはこれを変更しない**。変更が要ると判断したら止めて報告する。
- コミットは意味のある単位で行い、コミットメッセージにどのマイルストーン（M1〜M4）のどの受け入れ条件に対応するかを書く。

---

## 4. 完了報告のルール（重要）

- **「実装しました」だけの報告を完了とみなさない。** 各発注書末尾の**受け入れ条件YES/NO表**を自分で埋め、各項目に「YES＋根拠（どのファイルの何行がそれを満たすか）」を添えて報告する。
- 根拠を示せない項目は **YESにせず「未確認」と書く。** 未確認を隠さない。
- 検品はこの報告を人間（Claude）が別途照合して行う。あなたの自己申告だけで合格にはならない。

---

## 5. 質問の出し方

- 質問は「どうしますか？」ではなく、**具体的な二択／三択**にする。
  例：「写真0枚時のこの欄、A: 欄ごと隠す / B: 空表示文を出す。モック§6.4はBだが確認したい」
- 質問はまとめず、実装を止めた時点で1つずつ出す。
- 正本（モック・全体仕様書・技術決定書）に答えが書いてある質問はしない。まず読む。
